Get started with Grading System API

This API allows student portals to request grade details from the grading system using student unique ID, school year (SY), semester (SEM), year level, and section parameters. You can test API integrations with our developer tools.

Parameter Reference Tables

The API uses ID-based parameters for efficient database operations. Here are the reference mappings:

School Year IDs

school_year_id=1 2023-2024
school_year_id=2 2024-2025
school_year_id=3 2025-2026
school_year_id=4 2026-2027
school_year_id=5 2027-2028
school_year_id=6 2028-2029
IDs auto-increment as new school years are added by admin

String Parameters

Semester:
Use string values: "1st", "2nd", "Summer"
Year Level:
Use string values: "1st Year", "2nd Year", "Grade 11", etc.
Section:
Use string values: "HRM-A", "IT-B", "BSBA-C", etc.
Note: String parameters are flexible and can vary by school. Each institution can use their own naming conventions.

Why Use IDs for School Years Only?

‚Ä¢ School Year Efficiency: IDs solve database efficiency concerns
‚Ä¢ Student Retakes: Easy to handle students who repeat subjects across different years
‚Ä¢ String Flexibility: Semesters, year levels, and sections can vary by school
‚Ä¢ Simple Implementation: Only one ID table to manage (school years)
‚Ä¢ Auto-Increment: New school years automatically get the next ID (7, 8, 9...)
‚Ä¢ Best of Both: Efficient where needed, flexible everywhere else
Mixed Approach Benefits:
‚Ä¢ School Year ID: Addresses database efficiency concerns
‚Ä¢ String Parameters: No admin setup needed for semesters, levels, sections
‚Ä¢ Flexible for Schools: Each institution can use their own naming conventions
‚Ä¢ Easy Integration: Developers can use familiar string values

Recommended: Server-Side Proxy Implementation

For maximum security, implement a server-side proxy to hide API keys from client-side code. This prevents students or developers from seeing sensitive authentication headers in browser developer tools.

Security Risk: Client-Side API Keys

Problem: When API keys are used in client-side JavaScript, they become visible in browser developer tools, allowing unauthorized access.
// ‚ùå AVOID: Client-side API calls (API key visible in browser) fetch('https://api.edumanage.ph/v1/students/HRM2023-001/grades', { headers: { 'X-API-Key': 'sk_live_abc123xyz789secretkey' // ‚Üê Students can see this! } });
Risk: Students can copy API keys from developer tools and misuse them.

Solution: Server-Side Proxy

Recommended: Create a server-side proxy that handles API authentication while keeping keys secure.
Client-Side Code (Safe):
// ‚úÖ RECOMMENDED: Call your own server endpoint fetch('/api/student-grades', { method: 'POST', headers: { 'Content-Type': 'application/json' // ‚Üê Safe to show }, body: JSON.stringify({ studentId: 'HRM2023-001', schoolYearId: 1, semester: '1st' }) });
Server-Side Code (Hidden):
// ‚úÖ Server handles the real API call (API key hidden) app.post('/api/student-grades', async (req, res) => { const { studentId, schoolYearId, semester } = req.body; // API key stored securely on server const response = await fetch(`https://api.edumanage.ph/v1/students/${studentId}/grades`, { headers: { 'X-API-Key': process.env.SECRET_API_KEY, // ‚Üê Hidden from client 'Content-Type': 'application/json' } }); const grades = await response.json(); res.json(grades); });

Proxy Benefits

‚Ä¢ API keys completely hidden from browsers
‚Ä¢ Additional security layer and validation
‚Ä¢ Rate limiting and request monitoring
‚Ä¢ Centralized authentication management
